# Capture TUI 输入模块 - 需求分析报告

## 1. 模块概述

### 1.1 目标
设计一个基于 TUI (Terminal User Interface) 的输入捕获模块，用于收集、处理和输出用户的想法、任务和计划。

### 1.2 核心模块划分
```
┌─────────────────────────────────────────────────────────┐
│                    Capture TUI 系统                      │
├─────────────┬─────────────────────┬─────────────────────┤
│   Input     │      Process        │       Output        │
│   输入记录   │      处理模块        │      输出模块        │
├─────────────┼─────────────────────┼─────────────────────┤
│ • CLI命令   │ • 内容解析           │ • 项目目录输出       │
│ • 对话记录  │ • 任务提取           │ • 结构化导出        │
│ • 文件导入  │ • 计划生成           │ • 飞书多维表格      │
└─────────────┴─────────────────────┴─────────────────────┘
```

---

## 2. 场景分析

### 2.1 场景1：CLI直接输入

#### 使用场景描述
用户通过命令行直接输入待办事项、想法或任务，系统自动保存并进行分类管理。

#### 功能需求

| 序号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| 1.1 | 命令行输入支持 | P0 | 用户可直接输入文本内容 |
| 1.2 | 分类存储 | P0 | 按 category 保存到 ideas/ 目录 |
| 1.3 | 自动创建分类 | P1 | 无对应 category 时自动创建目录 |
| 1.4 | 分类数量限制 | P1 | 最多10个 category，超限提示 |
| 1.5 | 智能分析 | P2 | AI 分析 category 内容，提炼要点 |
| 1.6 | 文件路径支持 | P2 | 支持输入文件路径，读取内容 |
| 1.7 | 结构化导出 | P2 | 支持导出为 TODO Dashboard 或飞书表格 |

#### 数据结构
```
ideas/
├── category-1/           # 分类1
│   ├── 2024-01-15_001.md # 按日期编号存储
│   ├── 2024-01-15_002.md
│   └── summary.md        # 分类汇总分析
├── category-2/
│   └── ...
└── index.md              # 全局索引
```

#### 存储格式
```markdown
---
id: "ideas-20240115-001"
category: "feature-ideas"
created_at: "2024-01-15T10:30:00+08:00"
tags: ["ui", "improvement"]
source: "cli"
---

# 想法标题

具体内容描述...

## 关联任务
- [ ] 待分解为具体任务
```

### 2.2 场景2：Terminal 对话形式输入

#### 使用场景描述
启动交互式 CLI 会话，用户可以与 AI 进行对话，系统自动记录完整的对话过程和分析思路。

#### 功能需求

| 序号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| 2.1 | 对话记录捕获 | P0 | 记录完整对话过程 |
| 2.2 | 技能识别 | P1 | 识别 AI 使用的分析技能和思维方式 |
| 2.3 | 思考过程记录 | P1 | 记录 AI 的判断、分析、确认过程 |
| 2.4 | 元数据保存 | P2 | 保存使用的模型、技能、时间等 |
| 2.5 | 对话导出 | P2 | 导出为可查阅的格式 |

#### 技术选型对比

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| A | Skill 方式捕获 | 与现有工具集成，无需自建 | 依赖特定 CLI 工具 |
| B | 自建 TUI 对话 | 完全可控 | 重复造轮子，意义不大 |
| C | 日志拦截 | 通用性强 | 实现复杂 |

**推荐方案**: 方案 A (Skill 方式)
- 利用现有工具（claude code / kimi cli）的能力
- 通过 Skill 机制捕获对话记录
- 专注于记录和分析，而非重建对话系统

#### 记录内容规范
```markdown
---
session_id: "sess-20240115-001"
start_time: "2024-01-15T10:30:00+08:00"
end_time: "2024-01-15T10:45:00+08:00"
tool: "kimi-cli"
skills_used: ["explore", "plan", "coder"]
---

# 对话主题

## 用户目标
用户想要实现的功能/解决的问题

## 对话过程

### Turn 1
**用户**: 输入内容
**AI**: 回复内容
**思考过程**: AI 的分析思路
**使用技能**: explore

### Turn 2
...

## 技能使用分析
| 技能 | 使用次数 | 使用场景 |
|------|---------|---------|
| explore | 3 | 代码库探索 |
| plan | 1 | 架构规划 |

## 提取的行动项
- [ ] 具体任务1
- [ ] 具体任务2
```

  - 任务状态追踪

## 4. 关键问题与决策

### 4.1 如何存储便于后续查找？
**决策**: 
- 使用 YAML Frontmatter 元数据
- 全局索引文件记录所有条目
- 支持按 category、tag、时间检索

### 4.2 如何转化为结构化 TODO？
**决策**:
- 约定 Markdown 中的 TODO 标记语法
- 提供导出脚本转换为 CSV/JSON
- 支持飞书多维表格 API 对接

### 4.3 AI 分析如何实现？
**决策**:
- 作为独立命令: `capture analyze <category>`
- 使用 kimi-cli 的 Agent 能力
- 分析结果保存为 summary.md

### 4.4 对话记录如何完整捕获？
**决策**:
- 开发专用 skill: `conversation-recorder`
- 在 skill 中记录完整的输入输出
- 保存元数据（技能、模型、时间）

## 5. 风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| category 过多 | 管理混乱 | 强制10个上限 |
| 文件过多 | 检索困难 | 索引+搜索功能 |
| 格式不统一 | 处理困难 | 模板化+校验 |
| 对话记录过大 | 存储压力 | 定期归档 |
